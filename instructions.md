# HomePodMonitor Instructions

这个文件面向后续维护 HomePodMonitor 的开发者，目的是让改动保持一致，不要破坏当前菜单栏体验和音频切换策略。

## 项目目标

HomePodMonitor 的核心目标不是做一个完整音频管理器，而是稳定地把系统输出保持在用户选中的目标设备上，同时尽量减少对系统声音菜单的打扰。

任何改动都优先满足这两个要求：

- 用户能明确知道当前输出、默认输出、可切换输出之间的关系
- 应用不要因为后台检测而频繁弹出控制中心声音菜单

## 关键设计约束

### 1. 检测与切换必须分离

当前逻辑已经拆成两种路径：

- 被动检测：只通过 CoreAudio 判断当前默认输出，不主动点击菜单
- 主动切换：只有在确实需要切换时，才通过辅助功能操作控制中心声音菜单

修改 AudioDeviceController 时，不要把定时巡检重新改回“每轮都展开声音菜单读快照”的方式。当前策略是：目标正常时静默，只有检测到偏离默认输出时，巡检才允许主动切回。

### 2. 输出列表是唯一主交互入口

当前 UI 约束：

- 不再显示单独的“当前输出”区块
- 不再显示“立即切换到目标设备”按钮
- 用户通过列表项悬停后的“设为默认输出”按钮完成默认选择和立即切换

如果未来要增加功能，优先在列表项里扩展，而不是重新引入分散的独立按钮。

### 3. 颜色表达必须稳定

当前颜色语义如下：

- 默认输出：绿色
- 当前输出但不是默认输出：蓝色
- 其他输出：灰色，和该行图标颜色一致

不要随意改变这套语义，除非同时更新 README 并确认新的视觉语义更清晰。

### 4. 用户选择必须持久化

用户最后一次选择的默认输出通过 UserDefaults 持久化，键名是 preferredTargetName。

修改默认输出逻辑时必须保证：

- 选择后立即写入缓存
- 启动时优先从缓存恢复
- 没有缓存时再回退到“家庭影院”

## 文件职责

- HomePodMonitor/HomePodMonitorApp.swift
  负责 MenuBarExtra 生命周期和菜单栏图标状态

- HomePodMonitor/ContentView.swift
  负责菜单栏窗口 UI、输出列表呈现、悬停操作

- HomePodMonitor/AudioDeviceController.swift
  负责音频状态监听、默认输出缓存、切换节流、状态文案、开机启动

- HomePodMonitor/SoundOutputAccessibility.swift
  负责通过辅助功能读取控制中心声音菜单和执行切换

## 修改建议

### 改 UI 时

- 保持菜单栏窗口紧凑，不要做成常规设置页
- 输出列表的视觉优先级高于说明文案
- 如果增加新的状态提示，避免把主列表挤下去

### 改切换逻辑时

- 先判断是不是只需读当前输出，能不用 Accessibility 就不要用
- 只有必须改系统输出时，才展开控制中心声音菜单
- 保留当前的冷却时间和任务互斥，避免重复切换

### 改设备匹配时

- 当前主要按名称做大小写不敏感匹配
- 如果后续改成更稳定的标识符匹配，要保证 UI 仍展示人类可读名称

## 本地验证

常用构建命令：

```bash
xcodebuild -project HomePodMonitor.xcodeproj -scheme HomePodMonitor -configuration Debug build
```

手动验证时至少覆盖以下场景：

1. 首次启动，未授权辅助功能
2. 授权后读取到输出列表
3. 选择新的默认输出后立即切换成功
4. 重启应用后恢复上次默认输出
5. 定时巡检在已处于目标输出时不会主动弹开系统声音菜单；偏离目标后会自动触发切回
6. 当前输出和默认输出不一致时，颜色与状态文案正确
